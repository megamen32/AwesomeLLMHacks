# AwesomeLLMHacks

Подборка практических хаков, расширений, юзерскриптов и маленьких утилит для работы с LLM-сервисами: ChatGPT, Z.AI, DeepSeek, Qwen, MiniMax, Mistral, Grok, Google AI Studio и похожими веб-чатами.

Фокус репозитория — не «ещё один список ссылок», а прикладные штуки, которые реально помогают жить с веб-интерфейсами моделей: копировать ответы без мусора, видеть сетевой протокол, обходить раздражающие подтверждения, автоматизировать повтор запроса, держать несколько чатов рядом и превращать браузер в рабочий LLM-инструмент.

## Что сюда подходит

- Chrome/Chromium-расширения для LLM-сайтов.
- Tampermonkey/UserScript-хаки для ChatGPT-like интерфейсов.
- Инструменты для копирования, очередей, автоповтора, multi-pane, provider-hub и quota/status UI.
- Отладочные инструменты для анализа запросов, streaming responses, WebSocket и локальных браузерных состояний.
- Небольшие интеграции, которые закрывают конкретную боль в повседневной работе с LLM.

## Chrome-расширения

### [chatgpt-multi](https://github.com/megamen32/chatgpt-multi) — несколько рабочих панелей ChatGPT рядом

Chrome-расширение для многопанельного рабочего пространства ChatGPT: side-by-side чаты, lazy panes, очередь промптов, goal-agent и Telegram bridge. Это не просто «открыть несколько вкладок», а попытка сделать из ChatGPT нормальный рабочий стол для параллельной работы с несколькими контекстами.

**Зачем нужно:**

- сравнивать ответы разных моделей или разных промптов;
- держать рядом «план», «код», «проверку» и «финальный ответ»;
- не прыгать между вкладками, когда задача распадается на несколько параллельных мыслительных потоков;
- использовать ChatGPT как рабочее пространство, а не как одну линейную переписку;
- гонять подготовленные промпты через очередь, не теряя общий рабочий контекст.

**Типичный сценарий:**

1. Открываешь ChatGPT.
2. Расширение организует несколько панелей/рабочих областей.
3. В одной панели идёт основной диалог, в другой — проверка, в третьей — альтернативная версия или черновик.
4. Можно быстрее сводить результат, потому что контексты не перемешиваются.

**Почему это важно:** стандартный веб-интерфейс LLM почти всегда линейный. Для сложной работы с кодом, текстами, research и сравнением моделей удобнее многопанельный режим.

### [auto-confirm-extension](https://github.com/megamen32/auto-confirm-extension) — автоподтверждение action-dialogs в Custom GPT

Chrome-расширение, которое автоматически подтверждает диалоги действий ChatGPT Custom GPT. Особенно полезно для отладки и автоматизации, когда Custom GPT постоянно спрашивает подтверждение на одинаковых action-вызовах.

**Зачем нужно:**

- убрать ручную рутину в длинных агентных задачах;
- не сидеть перед экраном ради очередного одинакового подтверждения;
- ускорить работу с Custom GPT actions;
- сделать браузерный LLM-workflow ближе к нормальному batch/agent режиму.

**Важно:** такое расширение нужно использовать аккуратно. Автоподтверждение удобно там, где действие заранее ожидаемо и безопасно. Для опасных операций — отправка денег, удаление данных, публикация, выдача секретов — автоматический клик лучше не включать.

**Типичный сценарий:**

1. Custom GPT вызывает action.
2. ChatGPT показывает confirmation dialog.
3. Расширение распознаёт этот диалог и подтверждает его.
4. Агентная задача продолжает выполняться без ручного клика на каждом шаге.

### [promqueue](https://github.com/megamen32/promqueue) — очередь и массовая отправка промптов в LLM-чаты

Chrome-расширение для очереди и bulk-send промптов в z.ai, ChatGPT, Claude, DeepSeek, Qwen, Gemini, AI Studio и Copilot. Это более общий вариант идеи `zai-chat-queue`: не только Z.AI, а очередь промптов для разных веб-чатов.

**Зачем нужно:**

- ставить несколько сообщений/промптов в очередь;
- отправлять пачку задач без ручного ожидания каждого ответа;
- превращать веб-чаты в более предсказуемый рабочий конвейер;
- удобно гонять серии однотипных промптов: переписывание, генерация вариантов, анализ фрагментов, обработка списка задач;
- использовать один queue-подход для нескольких LLM-сервисов.

**Типичный сценарий:**

1. Пользователь готовит несколько промптов.
2. Расширение отправляет их последовательно или пачкой в выбранный веб-чат.
3. Если чат занят или ответ ещё генерируется, следующий запрос ждёт своей очереди.
4. В итоге веб-LLM можно использовать не только интерактивно, но и как полуавтоматический batch-инструмент.

**Почему это полезно:** веб-чаты часто не умеют нормальные очереди. Для реальной работы это боль: приходится руками помнить, что ещё надо отправить, и ловить момент, когда модель закончила отвечать.

### [zai-auto-retry](https://github.com/megamen32/zai-auto-retry) — автоповтор запроса, когда Z.AI показывает peak-hours dialog

Chrome-расширение, которое автоматически повторяет Z.AI chat request, когда появляется peak-hours dialog. Это узкий, но очень практичный hack для ситуации, когда запрос нормальный, но сервис временно перегружен.

**Зачем нужно:**

- не нажимать retry вручную;
- переживать временные перегрузы Z.AI без потери задачи;
- держать рабочий поток, когда сервис нестабилен;
- автоматически дожимать запрос, который почти наверняка пройдёт при повторе.

**Типичный сценарий:**

1. Пользователь отправляет запрос в Z.AI.
2. Z.AI показывает peak-hours/перегрузочный диалог.
3. Расширение ждёт и повторяет попытку.
4. После успешного ответа автоповтор прекращается.

**Практический смысл:** это маленький, но очень ценный hack для платных/лимитных веб-моделей, где проблема часто не в промпте, а во временной занятости сервиса.

## Tampermonkey/UserScript

### [Dynamic Network Interceptor — Full Stream Support](https://gist.github.com/Kreijstal/3b68d94c54e98a22b733e0fe0c739afd#file-fun-user-js)

Tampermonkey-скрипт для перехвата сетевой активности веб-интерфейсов ИИ.

Подходит для:

- Google AI Studio
- Mistral Chat
- Meta AI
- DeepSeek Chat
- Grok / xAI
- MiniMax Chat
- других похожих веб-чатов

**Что даёт:**

- видимость fetch/XHR/WebSocket-запросов;
- отладку streaming responses;
- анализ форматов запросов и ответов;
- наблюдение за тем, какие данные отправляются наружу;
- базу для reverse engineering и автоматизации LLM-интерфейсов.

**Главная идея:** скрипт делает фоновую сетевую активность выбранных ИИ-сайтов видимой для пользователя и позволяет реагировать на эти события прямо из консоли браузера.

Скрипт предоставляет объект `window.interceptorControl`:

- по умолчанию хуки отключены через `disable()`;
- `enableDefaultLogging()` включает подробный вывод в консоль;
- можно назначать собственные обработчики на события вроде `onFetchResponse`, `onFetchStreamChunk`, `onXhrStreamChunk`, `onWebSocketMessage`.

Пример идеи обработчика:

```js
window.interceptorControl.onFetchStreamChunk = (chunkText, url, isDone) => {
  if (chunkText) console.log('Новый кусок стрима:', chunkText);
};
```

Это фактически API для сетевых событий внутри страницы: можно собирать статистику, сохранять ответы, искать конкретные поля, строить свои overlay, логировать протокол или быстро проверять, что именно делает веб-чат.

### [GPT-AI-Markdown-LaTeX-Copier](https://github.com/Wavesflow/GPT-AI-Markdown-LaTeX-Copier/blob/main/UserScript%20for%20tampermonkey)

Tampermonkey-скрипт для копирования ответов ИИ в чистый Markdown с более нормальной поддержкой LaTeX-формул.

**Зачем нужно:**

- копировать ответы без лишнего HTML и мусора;
- сохранять структуру Markdown;
- лучше переносить формулы в заметки, статьи, README, Obsidian, Markdown-редакторы;
- не чинить вручную каждый математический блок после копирования.

## Провайдеры, роутинг и подписки

### [OmniRoute](https://github.com/diegosouzapw/OmniRoute) — все подписки и API-ключи к LLM в одном месте

[OmniRoute](https://github.com/diegosouzapw/OmniRoute) — идея единого слоя поверх разных LLM-провайдеров, подписок и API-ключей. Один endpoint на 230+ провайдеров (включая десятки бесплатных), RTK-компрессия токенов, auto-fallback, MCP/A2A, multimodal API и десктоп/PWA-клиент.

**Зачем нужно:**

- держать ключи и подписки централизованно;
- переключать модели без переписывания каждого клиента;
- строить `auto/best-coding`, `auto/best-coding-fast` и похожие профили;
- выбирать модель по цене, лимитам, качеству, доступности и задаче;
- не зависеть от одного провайдера, когда он лежит, режет лимиты или меняет правила.

**Практический смысл:** LLM-инфраструктура быстро превращается в зоопарк: ChatGPT, Claude, Z.AI, MiniMax, OpenRouter, GitHub Copilot, локальные модели, свои прокси. Нужен единый слой, который прячет этот хаос и даёт нормальный рабочий интерфейс.

## 📚 Утечки системных промтов

### [asgeirtj/system_prompts_leaks](https://github.com/asgeirtj/system_prompts_leaks/tree/main) — коллекция реальных system prompts популярных LLM

Большая курируемая коллекция утёкших и извлечённых системных промтов от основных провайдеров. Полезна как референс, когда хочется понять, **как именно** модель сконфигурирована изнутри: роли, tool-use, ограничения, форматы вывода, поведение в edge-cases.

**Что внутри:**

- Anthropic — Claude Opus, Sonnet, Claude Code, Claude Design.
- OpenAI — ChatGPT (Thinking / Instant), Codex.
- Google — Gemini 3.5 Flash, 3.1 Pro, Antigravity.
- xAI — Grok.
- Cursor, GitHub Copilot, VS Code-интеграции.
- Perplexity, Qwen, Mistral, Meta, Microsoft, Notion и др.

**Зачем это нужно в списке хаков:**

- быстро посмотреть, как модель на самом деле настроена, а не догадываться по поведению;
- использовать реальные промты как базу для собственных user/system префиксов в `promqueue`, `zai-auto-retry` и похожих инструментах;
- понимать, какие инструкции провайдер закладывает в Copilot/Code/Codex, чтобы делать более точные override-хаки;
- находить тонкие места в системных инструкциях (форматирование, refusals, tool routing) и строить на них собственные обходы.

**Как пользоваться:** репо обновляется регулярно. Самый быстрый вход — открыть нужную папку провайдера и сравнить несколько версий модели подряд. Это сразу видно, что менялось между релизами и куда уехали инструкции.

## Идеология списка

Хороший LLM-hack должен:

- закрывать реальную ежедневную боль;
- быть маленьким и понятным;
- не требовать тяжёлой инфраструктуры там, где достаточно браузерного расширения;
- помогать пользователю контролировать модель, а не наоборот;
- делать веб-интерфейсы LLM более автоматизируемыми, наблюдаемыми и переносимыми.
